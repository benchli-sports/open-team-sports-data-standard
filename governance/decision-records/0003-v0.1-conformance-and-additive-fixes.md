# 0003 - v0.1 Conformance Fixes and Additive Enhancements

## Status
Accepted

## Context
The benchli backend team made benchli a real, schema-validated OTSD
producer/consumer (validating every exported record and the manifest against the
vendored v0.1 schemas) and reported a set of issues found in practice
(2026-06-25). Two were blocking: the published v0.1 schemas were internally
**unsatisfiable**, so the project's own example export package failed the
project's own schemas. The remainder were model gaps that forced producers into
lossy mappings or extension workarounds.

This record covers the corrections and additive changes made in response. All
changes keep the `v0.1` label: because the blocking defects made conformant
`game`/`stat_line` records impossible to produce, there are no valid v0.1
packages in the wild to break, so the schemas are corrected in place rather than
forking a `v0.2/` tree.

## Decision

### Blocking — schema conformance fixes

1. **`game.json` could never validate (`type` const conflict).**
   `game.json` composes `event.json` via `allOf`, but `event.json` pinned `type`
   to const `"event"` while `game.json` pinned it to const `"game"` — no single
   value satisfies both. **Fix:** `event.json` `type` is now
   `enum: ["event", "game"]`, allowing concrete specializations to set their own
   `type`. The example `game` records also now include the required `category`.

2. **`stat_line.json` rejected integer metrics (`oneOf` ambiguity).**
   `metrics.additionalProperties` used `oneOf: [number, integer, …]`; an integer
   matches both `number` and `integer`, so `oneOf` (exactly-one) failed for every
   integer stat. **Fix:** changed to `anyOf` and dropped the redundant `integer`
   branch (`number` already covers integers).

### High — version field naming unified

3. **`otsd_version` (manifest) vs `schema_version` (record envelope).**
   Two names for the same "v0.1" concept in one package was a footgun. **Fix:**
   the manifest field is renamed `otsd_version` → `schema_version`, so
   `schema_version` is the single version field across the manifest and every
   NDJSON record envelope. This is a breaking rename, accepted under the pre-1.0
   policy (0.x permits breaking changes) and cheap because no conformant v0.1
   packages could previously be produced.

### Medium / Minor — model relaxations and additive fields

4. **`event.end_at` is now optional and nullable.** Grassroots events/games
   often have no recorded end time; producers no longer need to fabricate one.

5. **`game.category` is now `"game"` (was `"other"`).** Added `"game"` to the
   `event.category` enum and set `game.category` to const `"game"`, so a game is
   distinguishable by category and a real "other" event is no longer conflated
   with a game.

6. **`roster_snapshot.game_id` is now optional.** Allows team-/season-scoped
   roster snapshots that are not tied to a specific fixture.

7. **`appearance.minutes` accepts `number`** (was integer-only), for sports that
   track partial minutes/seconds.

8. **`payment_record` gains optional `request_id`, `parent_id`, `event_id`** to
   link a payment to the dues request/invoice it settles and to an event it
   funds, instead of stashing the link in a vendor extension.

9. **Manifest `scope.player_ids` (optional)** so a personal, player-scoped
   portability export can declare its subject. `team_ids` remains required for
   now; relaxing it when `player_ids` is present is possible future work.

### Additive — new `Availability` (RSVP) entity

10. **`availability` becomes a first-class optional entity** so participation
    *intent* is no longer second-class. A record ties `player_id` + `game_id` to
    a `status` (`yes`/`no`/`maybe`/`unknown`), with optional `responded_at` and
    `note`. `game.participation_summary` remains as a denormalized, possibly
    stale rollup that producers MAY derive from `availability` records;
    `availability` preserves the per-player timestamps and notes the summary
    loses.

### Tooling

11. **Validation made executable.** `scripts/validate.js` (AJV 2020-12) validates
    every example payload and the example export package against the schemas, run
    by `npm test` and in CI. The known-failure baseline that previously documented
    defects 1 and 2 is now empty.

## Consequences

### Positive
- The standard's own examples validate against its own schemas (26/26 cases).
- benchli can drop its in-memory schema corrections and emit real, conformant
  games and stats.
- Intent vs actual is now symmetric (first-class `availability` alongside
  `appearance`); fewer producers need extension workarounds.

### Neutral / Migration
- Items 4–10 are additive or relaxing and do not invalidate previously valid
  records.
- Item 3 (`otsd_version` → `schema_version`) is the one breaking change.
  Producers must rename the manifest field; consumers keyed on `otsd_version`
  must read `schema_version`. During 0.x, implementers MAY accept the legacy
  `otsd_version` as a deprecated alias to ease transition.

### Risks
- Widening `event.type` to an enum means `event.json` alone will accept a
  `game`-typed record; in practice records are validated by their specific
  schema (selected via the NDJSON envelope `type`), so the looser base schema is
  acceptable.
